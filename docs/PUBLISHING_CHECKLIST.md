# Publishing Checklist (dev.to & Hashnode)

Use this when posting **docs/ARTICLE.md** to dev.to or Hashnode.

## Title (SEO-friendly)

**Option A:**  
Build an End-to-End DevOps Pipeline (CI/CD + Docker + Kubernetes) — Zero Cloud Cost

**Option B:**  
How I Built a Portfolio-Ready DevOps Pipeline with GitHub Actions, Docker, and Kubernetes (Free)

## Subtitle / Description (for SEO and preview)

- dev.to: use the first paragraph or a 1–2 sentence summary in the post body; the first 150 chars often show in cards.
- Hashnode: set "Subtitle" to something like:  
  "A practical, portfolio-ready pipeline: GitHub Actions → Docker → GHCR → Kubernetes. No cloud spend required."

## Tags (both platforms)

- `devops`
- `cicd`
- `github-actions`
- `docker`
- `kubernetes`
- `terraform`
- `portfolio`
- `tutorial`

Use 4–5 on dev.to (max allowed), and equivalent on Hashnode.

## Before you publish

1. **Replace repo placeholder**  
   In the article, replace `[your-repo-url]` with your actual repo (e.g. `https://github.com/yourusername/devops-e2e-pipeline`).

2. **Canonical URL**  
   If you post on both dev.to and Hashnode, set the first post as canonical and the second can point to it (or vice versa) to avoid duplicate content issues.

3. **Code blocks**  
   The article uses markdown code blocks; dev.to and Hashnode will render them. No need to change unless you want syntax highlighting (usually automatic).

4. **Cover image (optional)**  
   Use a simple diagram (e.g. “repo → Actions → Docker → K8s”) or a standard DevOps image; both platforms support a cover image.

## Series (optional)

If you plan more posts (e.g. “Adding Argo CD” or “Terraform on AWS”), you can create a series like “DevOps portfolio projects” and link this as part 1.
